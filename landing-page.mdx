# LANDING PAGE DOCUMENTATION

## 1. PAGE OVERVIEW

- **Filename/Route:** [landing-page.component.ts](apps/bank-account-dao/src/app/landing-page/landing-page.component.ts) / Route: `/`
- **Description:** The Landing Page serves as the entry point for users in the Bank Account DAO application. It is the initial workflow selection screen where users choose their primary task: opening a new account, creating/editing a Customer Information File (CIF), or performing account maintenance. Based on the selected task and account type (Personal or Business), the page dynamically renders conditional form fields to guide users through the appropriate downstream workflow.
- **Feature Flags:**
  - `selectIsCifCreateEnabled`: Controls visibility of the "CREATE/EDIT A CIF" task option. When false, only "OPEN AN ACCOUNT" displays. When true, the user sees both options.
  - `selectIsAccountMaintenanceEnabled`: Controls visibility of the "MAINTENANCE ACCOUNT" task option. When true, the user can select this task for account maintenance workflows.

### Use Cases

**Use Case 1: Open a Personal Account (Default Flow)**

1. User lands on the page; feature flags are still loading.
2. If no feature flags are enabled, "OPEN AN ACCOUNT" task is auto-selected.
3. User selects "Personal" account type via button toggle.
4. User selects a Community Bank Location (e.g., "CHICAGO").
5. Ownership Type dropdown populates with available structures (e.g., "Individual").
6. User selects the ownership type.
7. User clicks "Continue".
8. The service dispatches `createEngagementAndSelectProfile` action and opens a customer search dialog.
9. User searches for and selects a personal customer.
10. Upon successful selection, router navigates to the customer overview page or next workflow step.

**Use Case 2: Create/Edit a Business CIF**

1. Feature flag `selectIsCifCreateEnabled` is true.
2. User selects "CREATE/EDIT A CIF" from task selector.
3. User selects "Business" account type.
4. Community Bank Location field is disabled (not required for CIF-only flow).
5. Entity Type dropdown populates (using default bank location internally).
6. User selects an entity type (e.g., "LLC").
7. User clicks "Continue".
8. The service dispatches `createEngagementAndSelectProfile` action and opens a business search dialog.
9. User searches for and selects a business.
10. If business has a CIF, profile selection succeeds and dialog closes. If no CIF exists, a new CIF creation flow is initiated.
11. Router navigates to customer overview.

**Use Case 3: Account Maintenance**

1. Feature flag `selectIsAccountMaintenanceEnabled` is true.
2. User selects "MAINTENANCE ACCOUNT" from task selector.
3. Account Type and Location fields are disabled.
4. User clicks "Continue".
5. The service opens an Account Maintenance search dialog.
6. User searches for and selects an account (individual or business).
7. Upon successful selection, router navigates to the customer overview page.

### Conditions & Edge Cases

| Condition | Behavior |
| --- | --- |
| **Feature flags not loaded** | Page displays normally but waits for `selectFeatureFlagsLoaded` selector to emit true before auto-selecting default task. |
| **Ownership structures loading** | Skeleton loader displays in place of the form card. Continue button is disabled. `structuresLoading()` = true. |
| **Ownership structures API fails** | `structureSets()` remains null, skeleton continues to display indefinitely (no explicit error state in component). |
| **Form validation fails** | Continue button is disabled. User must correct form errors (missing required selections). |
| **User selects Account Type = Business in CIF flow** | Entity Type dropdown is enabled instead of Ownership Type. Only business entity structures are available. |
| **Only one ownership type available** | Dropdown automatically pre-fills and marks field as untouched (no user input required). |
| **User changes Community Bank Location** | Ownership Type dropdown resets, fetches options for new location, and auto-fills if only one option exists. |
| **User changes Account Type while Community Bank Location selected** | For CIF Business flow: Entity Type options update automatically. For Open Account: Ownership Type options update. |
| **Customer/Business search dialog closed without selection** | Dialog closes, page remains on landing page with form state preserved. No navigation occurs. |
| **Customer/Business creation fails** | Error actions are dispatched; dialog remains open; no navigation occurs. |
| **User not authenticated** | `samAccountName` and `relationshipBankerName` are undefined; actions are still dispatched with these undefined values (backend may handle gracefully). |

## 5. REACTIVE PATTERNS

### Reactive Streams

1. **Feature Flags Loading Stream**
   - **Description:** On component initialization, the feature flags are reloaded. The component waits for `selectFeatureFlagsLoaded` to emit true (using `filter()` and `take(1)`). Once flags are loaded and if no toggle flags are enabled, the default task "OPEN AN ACCOUNT" is auto-selected.
   - **RxJS Operators:** `select()`, `filter()`, `take()`, `takeUntilDestroyed()`
2. **Ownership Structures Loading Stream**
   - **Description:** After the component loads, it waits for `ownershipStructureFeature.selectOwnershipStructures` to emit a non-null value. Once data arrives, the `_isFetchingOptions` signal is set to false.
   - **RxJS Operators:** `select()`, `filter()`, `take()`, `takeUntilDestroyed()`
3. **Task Selection Changes Stream**
   - **Description:** Whenever the user changes the selected task (via button toggle), the component updates flow flags (`isOpenAccountFlow`, `isCreateCifBusinessFlow`, `isAccountMaintenanceFlow`), dispatches UI flow actions (`setCreateCifOnly`, `setIsAccountMaintenance`), and re-enables/disables form controls.
   - **RxJS Operators:** `valueChanges`, `takeUntilDestroyed()`
4. **Account Type Changes Stream**
   - **Description:** When account type changes, the component updates the stored `accountType` and `searchType`, calls `updateFlowFlags()`, and (for CIF Business flow) fetches and auto-fills ownership types.
   - **RxJS Operators:** `valueChanges`, `takeUntilDestroyed()`
5. **Combined Location & Account Type Changes Stream**
   - **Description:** A `combineLatest` stream watches both `communityBankControl.valueChanges` and `accountTypeControl.valueChanges`. When both have values, it fetches ownership structures for the selected location and account type. If only one structure exists, it auto-fills the ownership type dropdown.
   - **RxJS Operators:** `combineLatest()`, `filter()`, `takeUntilDestroyed()`
6. **Form Value Stream**
   - **Description:** The `formValue` signal is derived from `form.valueChanges` using `toSignal()`. It provides the current raw form value and updates reactively whenever any form control changes.
   - **RxJS Operators:** `valueChanges`, `map()`, `toSignal()`

### Data Life Cycle

- **OnInit:** Component dispatches `landingPageComponentActions.loaded()` and `featureFlagsActions.reload()` to initialize state. It sets up all listener subscriptions and waits for feature flags to load.
- **Form Changes:** Any form control change triggers appropriate listeners that update flow flags, dispatch UI flow actions, and optionally fetch new ownership structures data.
- **OnDestroy:** The `DestroyRef` is used with `takeUntilDestroyed()` to automatically unsubscribe from all streams when the component is destroyed. This prevents memory leaks.
- **State Cleanup:** Before continuing to the next workflow, the service calls `clearSearchData()` to clear previous ownership structure data via `engagementHydrationServiceOwnershipActions.clearState()` and `SearchUtils.clearSearchData()`.

### Effects

Effects that respond to Landing Page actions:

- **Feature Flags Effects:** Listen to `featureFlagsActions.reload()` and fetch feature flag data from the backend
- **Ownership Structures Effects:** Listen to `landingPageComponentActions.loaded()` and fetch ownership structures
- **Engagement Effects:** Listen to `landingPageComponentEngagementActions.createEngagementAndSelectProfile()` and orchestrate engagement and profile selection workflows
- **Profile Selection Effects:** Listen to profile selection actions and navigate to the appropriate screen (e.g., customer overview, account maintenance) or handle failures by opening creation dialogs
- **Business Orchestration Effects:** Listen to business-related actions and coordinate business creation, profile selection, and account opening workflows

### Reducers

**Landing Page Component Reducer:**

- Handles `landingPageComponentActions.updateOwnership()`: Stores selected ownership structure details (accountType, ownershipTypeId, classCode, isPod, isRos, etc.)
- Handles `landingPageComponentActions.loadingCustomerProfile()`: Updates loading state
- Handles `landingPageComponentActions.addCustomer()`: Stores customer data
- Handles `landingPageComponentActions.addCustomers()`: Stores multiple customers
- Handles `landingPageComponentActions.setBusinessLoading()`: Updates business loading state

**Feature Flags Reducer:**

- Handles `featureFlagsActions.reload()`: Triggers loading state
- Stores feature flag values (`selectIsCifCreateEnabled`, `selectIsAccountMaintenanceEnabled`) based on backend response

**Ownership Structures Reducer:**

- Handles fetch actions and stores the returned ownership structures data

### Performance Notes

1. **Change Detection Strategy:** Not explicitly set (defaults to default strategy). The component uses a mix of:
   - **Signals** for fast, fine-grained reactivity: `structureSets`, `isCifCreateEnabled`, `isAccountMaintenanceEnabled`, `_isFetchingOptions`
   - **Computed signals** for derived state: `taskOptions()`, `availableStructures()`, `structureOptions()`, `structuresLoading()`. These recompute only when their dependencies change.
   - **toSignal()** for converting RxJS streams to signals: `formValue`
2. **Heavy Computations:**
   - **Ownership Structures Filtering** (`availableStructures()` computed): Filters ownership structures by account type and state code. This is efficient if the data set is small but could be optimized in the backend if the data grows large.
   - **Structure Options Mapping** (`structureOptions()` computed): Maps available structures to dropdown options on every change. Runs synchronously but should be lightweight.
3. **Async Operations:**
   - **`getOwnershipStructures()`**: Makes an HTTP request to the backend via `OwnershipTypesService`. The component sets `_isFetchingOptions` to true during the fetch and false upon completion. The UI disables the Continue button and related dropdowns during the loading state.
   - **`onContinue()` async method**: Handles multiple workflows (personal, business, maintenance) with dialog subscriptions and action dispatches. All subscriptions are properly cleaned up via `takeUntilDestroyed()`.
4. **Memory Management:**
   - All RxJS subscriptions are properly cleaned up using `takeUntilDestroyed(this.destroyRef)`. This prevents memory leaks when the component is destroyed.
   - Reactive form listeners via `valueChanges` are also managed with `takeUntilDestroyed()`.
5. **Optimization Opportunities:**
   - Consider OnPush change detection strategy if the component becomes part of a larger application to reduce unnecessary change detection cycles. Currently uses the default strategy.
   - Ownership structures could be cached or preloaded at the app level rather than fetched on demand for each location/account type combination.
   - The `formValue` signal using `toSignal()` could be replaced with direct form control subscriptions if the current reactive pattern becomes performance-critical.
